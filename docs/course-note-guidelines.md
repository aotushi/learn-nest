# Course Note Guidelines

These rules exist to reduce information loss when turning a course section into local notes.

## Priority

Course notes must preserve the course learning path first, then add local clarification.

Write in this order:

```text
course flow
local reproduction
version differences
extra understanding
real project advice
```

Do not replace the course structure with a broader official-documentation structure unless the note clearly separates it as extra context.

## Required Coverage

For each lesson, capture the items that appear in the course:

- CLI commands shown in the lesson.
- Files generated by those commands.
- Initial generated shape when it matters.
- Files changed during the lesson.
- Request URL, input, output, and console output when the course demonstrates runtime behavior.
- Errors encountered locally and the exact fix.
- Version differences between course code and local dependencies.

Before drafting a note, make a course example inventory:

```text
commands
generated files
edited files
decorators or APIs introduced
request paths
response body
response headers
console output
local errors
required setup
```

Every course example in the inventory must have a corresponding place in the note.

## Mechanism Notes

When a lesson introduces a framework mechanism, use this structure:

```text
what it is
creation command
generated file shape
core method or decorator
how it is enabled
where it takes effect
trigger condition
common pitfalls
```

If the example needs setup before the mechanism works, put that setup beside the example.

Examples:

```text
session middleware before @Session()
FileInterceptor before @UploadedFile()
static assets directory, views directory, and view engine before @Render()
host matching before @HostParam()
CORS setup before cross-origin frontend requests
```

For Nest AOP-related mechanisms, use the same pattern consistently:

```text
Middleware
Guard
Pipe
Interceptor
Exception Filter
```

## Course Categories First

If the course uses a specific classification, keep that classification in the note.

Example:

```text
global usage
route usage
module provider usage
```

Do not replace it only with:

```text
global level
controller level
method level
parameter level
```

The second form can be added as a summary, but it must not erase the course's own flow.

## Version Differences

When course code does not work locally, record:

```text
course code
local error
local dependency versions
reason for the difference
current recommended code
```

Use concrete versions and dates when possible.

## Runtime Examples

When a lesson demonstrates a request/response, record:

```text
request URL
decorators involved
execution chain
source of the returned value or error message
final response
```

For example, if a Pipe throws and a Filter catches it, the note should identify which file creates the message and which file formats the response.

Runtime examples should live next to the concept they explain.

Prefer:

```text
@Res() explanation
  -> code
  -> request
  -> output
  -> source of response
```

Avoid putting the main explanation in a far-away catch-all runtime section.

A final runtime section is allowed only as a short self-test index:

```text
request path
concept
what to check
```

It should not contain the primary explanation.

When a Nest feature changes the default `return` behavior, explicitly identify the response source.

Examples:

```text
@Res(): response comes from res.send/json/end, not handler return.
@Next(): response may come from the next matching handler.
@Render(): handler return object becomes template data, not JSON.
@Session(): Set-Cookie is written by express-session, not by handler return.
```

## Minimal Reproduction

Each lesson note should include a small reproduction path when the lesson has runnable code:

```text
1. commands to run
2. files to edit
3. URL or command to test
4. expected result
```

## Keep Notes Useful

Avoid over-compressing a lesson into abstract concepts only.

Avoid over-expanding into a full official reference manual.

The goal is:

```text
enough detail to replay the course section locally without losing the course's order.
```

## Post-Write Audit

After writing or updating a lesson note, audit it against the course example inventory:

```text
1. Is every course command recorded?
2. Is every code example represented near the concept it teaches?
3. Is every request path recorded with expected output?
4. Is every response source clear?
5. Is every required setup recorded beside the feature that needs it?
6. Are local version differences and fixes recorded?
7. Is the final self-test section only an index, not the main explanation?
```
